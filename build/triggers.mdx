Triggers are how agents **wake up**. Instead of you starting each run, the agent starts itself when its trigger fires. Choosing the right trigger type is the single biggest lever you have on both cost and reliability.

## The three trigger types

| Type | The agent runs… | Best for | Watch out |
|---|---|---|---|
| **Manual** | when you start it | experiments, one-off jobs, sensitive workflows where a human decides every run | doesn't scale beyond you |
| **Scheduled** | on a fixed cadence (hourly, daily, weekly) | digests, briefings, genuinely time-based reports | usually not the best option — runs whether or not there's work |
| **Event-based** | only when something happens (Slack message, webhook, integration event) | reactive workflows — "when X happens, do Y" | design for missed events in very bursty scenarios |

The rule of thumb: **event-based triggers are often the best choice** — the agent runs only when there is actual work, and it reacts instantly. Scheduled triggers can work, and manual triggers have their place for workflows a human should start deliberately. But the rule that holds for every type: **make the trigger as precise as possible**.

<Info>
  **A real example:** a meeting-briefing agent first ran on a daily 8 AM schedule. Weekend runs found nothing, and meetings booked during the day were missed entirely. Rebuilt with an event trigger — wake up 15 minutes before each meeting — it runs only when there actually is a meeting, always with fresh information.
</Info>

<Note>
  **Cost example:** checking for new items on a tight schedule costs around $50 per month, because most runs find nothing. The same workflow with an event trigger costs around $0.50 per month. That is roughly 100x cheaper.
</Note>

## Make every trigger as specific as possible

Use trigger conditions so the agent does not run on irrelevant events — every filtered-out event is a run you don't pay for and an error that can't happen.

| | Trigger | What happens |
|---|---|---|
| 🔴 Costly | On **any new email** | Fires on everything and burns runs on noise. |
| 🟢 Lean | Filtered by **sender, subject & keywords** | Only the few relevant emails ever wake the agent. |

Specific triggers = fewer runs, lower cost.

## Manual triggers you already have

The best trigger is often a quick action you already do in your daily tools — no new infrastructure needed:

| Pattern | How it works |
|---|---|
| **Notion automations** | A property change (e.g., status set to "Done") fires a Dataleap workflow: Notion → Create automation → trigger on property edit. |
| **Labels as triggers** | In Gmail, HubSpot, Jira, and even ATS tools, applying a label to an item starts the agent — a powerful pattern most users don't think of. |
| **Any quick action** | Labels, status changes, checkboxes, tags: if it's a one-click action in an interface you already use, it can trigger an agent. |

## Approvals as triggers

Human-in-the-loop checkpoints can be built with triggers: the agent sets a field, status, or label and stops; a human changing that value is the event that triggers the next step. This gives you an approval gate without extra infrastructure — see [Shape agent behavior](/build/shape-agent-behavior) for the HITL pattern.

## Two things that bite

<Warning>
  **A copied agent carries the original's triggers** (disabled by default). Reconfigure them before enabling — otherwise the copy fires alongside the original.
</Warning>

<Warning>
  For very bursty event sources (many simultaneous webhook events), don't assume every event is queued and processed — runs can be dropped in burst scenarios. If your workflow can't tolerate a missed event, add a periodic catch-up run.
</Warning>
